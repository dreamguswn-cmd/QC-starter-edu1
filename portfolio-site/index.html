PROJECT 01

Rule vs API 동시 비교 QA 파이프라인

같은 질문-같은 기준-같은 순간에 두 챗봇(규칙 기반 / OpenAI API 기반)을 동시에 평가하는 통제 비교 시스템

PythonOpenAI Responses APIStructured OutputsStreamlit · AltairpandasJira · WBS
2인 라이브 시연

PROBLEM — 문제

AI 챗봇을 서비스에 붙일 때 현업의 첫 질문은 '비싼 API가 꼭 필요한가, 규칙 기반으로 충분한가?'입니다. 이 판단을 감으로 하면 근거가 남지 않습니다. PROJECT 05에서 단일 챗봇 QA 파이프라인을 만들면 "검증기 자체가 오탐을 낼 수 있다"는 것을 확인했기 때문에, 파이널 프로젝트에서는 한 단계 더 나아가 같은 질문-같은 기준-같은 순간에 두 챗봇(규칙 기반 / OpenAI API 기반)을 동시에 평가하는 통합 시스템으로 확장했습니다.

PROCESS — 해결 과정

  1. 검증기 재설계 (6건 FAIL 사전 해결) — 발표 전날 실행에서 7건 중 6건 FAIL(합격률 14.3%)이 발생. 검증기를 단순히 키워드 모의를 단서로 챗봇이 아닌 검증기를 의심, 자동 완전일치 요구·키워드 자동 추가·규칙/의미 동조라는 3종 과설계를 찾아내 '케이스별 기대 카드' 포함 검사'로 단순화
  2. 동시 비교 파이프라인 — main.py가 케이스 하나당 규칙 답변과 API 답변을 같은 루프 안에서 동시에 생성 → 검증 → 평가. 같은 조건이므로 점수 차이는 순전히 모델 차이로 해석 가능
  3. 통합 와이드 CSV 설계 — 케이스당 1행에 rule_base_* / api_base_* 컬럼 두 세트를 담고, 대시보드가 차트를 그릴 때 세로(long format)로 변환하며 모델 식별자(version)를 부여하는 구조로 전환. 전환기에 구형 CSV를 감지해 재실행을 안내하는 방어 코드 추가
  4. 비교 대시보드 고도화 — 모델 필터, 지표별 범주노드 점수 배치(🟢5 🟡4 🟠3 🔴1-2), 색박 다비 숫자 뱅기), 카테고리·장녹벽 Rule/API 그룹 띠다(Altair), 케이스 상세 좌우 비고, 그리고 장단점·점수차·종합의견까지 이터에서 자동 생성
  5. 발표·시연 설계 (2인 담 파트) — 발표자(멘트)와 조작자(화면)를 분리한 2인 큐시트를 설계해 멘트 한 줄마다 화면 조작을 1:1로 짝지어 배치. 수정 전(6건 FAIL) 대시보드 → 라인 코드 → 수정 후 라이브 실행으로 이어지는 Before/After 시연 동선과, QA 사전 질문 10개에 대한 코드 근거 기반 답변·화면 자체 방법 업 준비
  6. 프로젝트 관리 — Jira 칸반(총괄·개발 / QA·품질관리 / Dashboard / 발표·시연)과 WBS로 4인 역할·일정을 관리하고 리허설까지 완료

RESULT — 결과

7 : 7 PASS

Rule vs API 통과율 — 표현상 완전 동률 (약 100%)

4.82점목 평균 점수 (동률)
1점TC-006 유승 — 두 모달 공통 회독
14.3→100%검증기 재설계 전후 합격률

✓ 통과율이 같아도 품질이 같은 것은 아니다 — 7:7 동률에서도 표현의 자연스러움(Rule은 정형 조합, API는 자연 문장), 실행 간 변동성(같은 케이스 3→4점), 거절 품질 차이를 지표 단위로 분리해 읽어냈습니다.

✓ 발표 전날의 6건 승기지의 핵심 스토리로 전환했습니다 — 결함 발견 → 원인 분석 → 재설계 → 정상화 과정을 심사 질문(사전 질문 10개)에도 코드 근거를 달 수 있게 준비했습니다.

PROJECT 02

RAG 챗봇 자동 품질 평가 및 답변 개선 시스템

Judge Agent 평가 결과를 Correction Agent에 재입력해 실패 답변을 모범답안으로 자동 수정하는 피드백 루프

PythonOpenAI APIChromaDBLLM-as-a-JudgeJSON·Markdown

PROBLEM — 문제

RAG 챗봇을 검색이 잘 됩는자-답변이 근거에 맞는지-문서에 없는 내용을 지어내지(환각) 않는지를 사람이 매번 직접 확인하면 기준이 흔들리고 시간이 오래 걸립니다. 또한 점수만 매겨서는 낮은 점수를 받은 답변이 그대로 폐기되어, 평가가 실제 개선으로 이어지지 않는다는 한계가 있었습니다.

PROCESS — 해결 과정

  1. RAG 파이프라인 구축 — 문서 임베딩 → ChromaDB 저장 → 질문 시 유사 문서 검색 → 근거 기반 답변 생성
  2. 테스트 케이스 설계 — 정상·경계·환각·유도·안전 거절 등 15개 카테고리로 시나리오 구성
  3. Judge Agent 구현 (1단계 평가) — 이해도·정확성·관련성·표현성 4항목을 0~5점 루브릭으로 채점
  4. 감점 규칙 적용 (2단계 평가) — 출처 미기재·근거 미사용·정의만 재공 등 규칙 위반 시 점정 감점 → 최종점수 = 루브릭 − 경감 으로 점수 인플레이션 방지
  5. Correction Agent 구현 — 평가 리포트(JSON)를 입력받아 감점 원인을 반영한 수정 모범답안을 자동 생성
  6. 리포트 자동화 — 전체 검증을 JSON + Markdown 리포트로 자동 저장 (run_tests.py)

RESULT — 결과

4.40전체 평균 / 5.00 (15개 케이스)
15테스트 카테고리
5.00최고 점수
2.75최저 점수 → 개선

✓ Two-Stage Evaluation으로 평가 근거를 항상 설명 가능하게 설계했습니다.

✓ 평가에서 멈추지 않고 개선까지 이어지는 피드백 루프(Evaluation → Correction → Reporting)를 구현했습니다.

✓ Prompt Engineering, RAG, Vector DB를 함께 다루며 AI 서비스 품질관리 자동화를 실무 수준으로 경험했습니다.

PROJECT 03

BERTScore 기반 AI 답변 품질 평가 시스템

의미 기반 유사도로 챗봇 응답 품질을 자동 채점하는 QA 파이프라인

Pythonbert-scorePyTorchpandas

PROBLEM — 문제

교육기관 챗봇의 답변 품질을 사람이 매번 눈으로 확인하면 비효율적이고, 단순 문자열 비교(키워드 일치)는 문장 표현이 다르면 의미가 같아도 오답으로 판정하는 한계가 있었습니다. 의미 기반으로 정답 여부를 자동으로 판단할 평가 체계가 필요했습니다.

PROCESS — 해결 과정

  3. BERTScore 계산 — Golden Answer와 AI 답변(Model A, B) 간 의미적 유사도 산출
  4. Winner 자동 판정 — 두 모델 점수를 비교해 케이스별 우승 모델 자동 기록
  5. CSV 저장 & 평균 분석 — pandas로 결과를 누적·집계해 전체 성능 비교

RESULT — 결과

CASEMODEL AMODEL BWINNER
Q0010.98120.6804Model A
Q0020.93210.7102Model A

✓ 문자열 비교가 아닌 의미 기반 평가의 필요성과 원리를 이해했습니다.

✓ Golden Answer 설계가 평가 신뢰도를 좌우한다는 점을 학습했습니다.

✓ Python 기반 AI QA 자동화 파이프라인 구축 경험을 쌓았습니다.

PROJECT 04

기준 모델 기반 BERTScore 회귀 평가

새 모델(B)이 기존 모델(A)의 응답 특성을 유지하는지 배포 전에 자동 점검하는 회귀 테스트

Pythonbert-scorepandas

PROBLEM — 문제

Golden Answer 기반 평가는 "정답에 가까운가"를 보여주지만, 모델을 교체했을 때 기준 서비스 품질이 그대로 유지되는지는 별도로 확인해야 합니다. 새 모델이 더 효율적이거나 저렴하더라도, 기존 모델과 응답 방향이 달라지면 사용자 경험이 갑자기 바뀔 수 있기 때문입니다.

PROCESS — 해결 과정

  1. 기준 모델(A) 응답을 Reference로 설정 — 정답이 아니라 '기준 서비스가 보이던 답변'을 기준점으로 사용
  2. 비교 모델(B) 응답과 BERTScore 산출 — Precision/Recall/F1로 두 응답의 의미적 거리 측정
  3. 3단계 자동 판정 기준 설계 — F1 0.95↑ 안정적 유지 / 0.85↑ 검토 필요 / 0.85↓ 성능 저하 의심
  4. 케이스별 평가 의견 자동 생성 — 판정 결과에 맞는 QA 코멘트를 템플릿으로 자동 작성
  5. 종합 보고서 자동화 — 전체 F1, 판정 분포, 배포 권고 사항을 Markdown으로 출력

RESULT — 결과

0.797평균 F1
0이행없이 요인
1검토 타당
9성능 저하 의심

✓ BERTScore는 표현 차이에 민감해서, 의미가 같아도 점수가 낮게 나올 수 있다는 한계를 확인했습니다.

✓ 자동 점수는 1차 필터링일 뿐, 최종 판단은 사람의 검토와 결합해야 한다는 QA 원칙을 정리했습니다.

✓ 안전성·환각·숫자·금액처럼 자동 지표만으로 판단하면 안 되는 영역을 구분하는 기준을 세웠습니다.

PROJECT 05

Routing Classifier 기반 AI Agent 자동 분류 품질평가

사용자 질문을 올바른 전문 Agent로 연결하는 분류기의 정확도와 오분류 원인 분석

Pythonscikit-learnTF-IDFLogistic Regressionmatplotlib

PROBLEM — 문제

여러 전문 Agent(스포츠/타이머/주문/일반대화)가 있는 챗봇에서는, 질문에 잘 답하는 것보다 올바른 Agent로 먼저 연결되는 것이 우선입니다. "결제 방법 알려줘"가 스포츠 Agent로 잘못 연결된다면, 답변 품질과 무관하게 사용자는 원하는 도움을 받을 수 없습니다.

PROCESS — 해결 과정

  1. 분류기 학습 — TF-IDF(문자 2~5-gram) + Logistic Regression으로 4개 Agent 분류 파이프라인 구축
  2. 정상 케이스 평가 — 스포츠·타이머·주문·일반대화 16건으로 Accuracy, Precision/Recall/F1 측정
  3. 혼동행렬(Confusion Matrix) 분석 — 어떤 Agent끼리 혼동되는지 시각화
  4. Edge Case 설계 — 오타, 복합 의도, 의미 없는 입력, 짧은 문장 등 8건 별도 테스트
  5. 오분류 원인 분석 & 개선안 도출 — 데이터 불균형과 문맥 부족 문제를 구체적 개선 방향으로 정리

RESULT — 결과

93.75%정상 케이스 정확도 (15/16)
87.5%Edge Case 정확도 (7/8)
4분류 대상 Agent

✓ Accuracy 하나만 보지 않고 혼동행렬로 어떤 Agent 간 혼동이 발생하는지 분석하는 습관을 들였습니다.

✓ 정상 케이스만으로는 부족하며 오타·복합의도 등 Edge Case 테스트가 따로 필요하다는 것을 입증했습니다.

✓ 모델 성능 평가는 Accuracy뿐 아니라 Precision/Recall/F1과 오분류 사례 분석이 함께 이루어져야 한다고 체득했습니다.

PROJECT 06

AI 교육과정 안내 챗봇 품질관리 자동화 파이프라인

Rule Validator(1차 검증) + Judge Agent(2차 AI 평가)를 결합하고, 테스트 리포트·대시보드·모니터링까지 연결한 엔드투엔드 QA 시스템

PythonFastAPIOpenAI Responses APIpytestStreamlitPrometheus · GrafanaDocker Composek6

PROBLEM — 문제

교육과정 안내 챗봇은 교육시간·출결·수료 기준처럼 숫자 하나만 틀려도 사용자에게 직접적인 피해로 이어지는 정보를 다룹니다. 답변 품질을 사람이 매번 눈으로 확인하면 판변 기준이 흔들리고, 평가 과정을 기록해두지 않으면 같은 실수가 반복됩니다. 교육시간·출결 수료 기준 온데나, 기준 문서에 없는 질문에 대한 한함 답변, 위험하거나 부적절한 요청에 대한 부적절한 대응을 사람 손을 거치지 않고 재현 가능하게 잡아내는 것이 이 프로젝트의 목표였습니다.

PROCESS — 해결 과정

  1. FastAPI 서비스 설계 — GET /health, POST /evaluate, POST /pipeline/run, GET /metrics 4개 엔드포인트로 단일 질문 평가와 전체 파이프라인 실행을 모두 API로 노출
  2. Service Agent 구현 — 안전 완판 → 문서와 질문 → 읽고 있는 주제 순으로 반응하는 키워드 기반 모의 챗봇을 구현해, API 카가 없는 환경에서도 파이프라인이 항상 실행되도록 설계
  3. Rule Validator (1차 검증) — 답변에 예상 키워드가 포함됐는지 먼저 기계적으로 확인하는 규칙 기반 1차 안전방 구현
  4. Judge Agent (2차 검증) — OpenAI Responses API + Structured Outputs(JSON Schema, strict: true)로 정확성·근거성·유용성·안전성 4항목을 자동 채점, API 카가 없거나 API 규칙 호출이 실패하면 규칙 팔백 평가로 자동 전환
  5. 리포트 자동화 — 케이스별 평가 결과를 JSON·CSV·Markdown으로 자동 저장(report_generator.py)하고, Streamlit 대시보드로 PASS율·항목별 평균 점수·카테고리별 PASS/FAIL을 시각화
  6. 테스트 & 운영 확장 — pytest로 health/API(파이프라인/안전성) negative case 4개 테스트 스위트를 구성하고, k6 성능 테스트와 Prometheus·Grafana 모니터링, Docker Compose 배포까지 풀더 구조에 반영

PROJECT 07

RAG 기반 문서 챗봇 (팀 프로젝트)

등록된 문서 내용에 근거하여 답변하는 RAG 기반 문서 챗봇 — 4인 팀 프로젝트 (팀표 역할)

PythonRAGChromaDBOpenAI APILLM Judge

PROBLEM — 문제

RAG 챗봇을 검색이 잘 됩는지-답변이 근거에 맞는지-문서에 없는 내용을 지어내지(환각) 않는지를 사람이 매번 직접 확인하는 기준이 흔들리고 시간이 오래 걸립니다. 또한 점수만 매겨서는 낮은 점수를 받은 답변이 그대로 폐기되어, 평가가 실제 개선으로 이어지지 않는다는 한계가 있었습니다.

PROCESS — 해결 과정

  1. RAG 파이프라인 구축 — 문서 임베딩 → ChromaDB 저장 → 질문 시 유사 문서 검색 → 근거 기반 답변 생성
  2. 테스트 케이스 설계 — 정상·경계·환각·유도·안전 거절 등 15개 카테고리로 시나리오 구성
  3. Judge Agent 구현 (1단계 평가) — 이해도·정확성·관련성·표현성 4항목을 0~5점 루브릭으로 채점
  4. 감점 규칙 적용 (2단계 평가) — 출처 미기재·근거 미사용·정의만 재공 등 규칙 위반 시 감점
  5. Correction Agent 구현 — 평가 리포트(JSON)를 입력받아 수정 모범답안 자동 생성
  6. 리포트 자동화 — 전체 검증을 JSON + Markdown 리포트로 자동 저장
