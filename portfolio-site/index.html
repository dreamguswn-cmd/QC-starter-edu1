PROJECT 01 · FEATURED

RAG 챗봇 자동 품질평가 & 답변 개선 시스템

Judge Agent로 채점하고, Correction Agent로 실패 답변을 자동 개선하는 QA 피드백 루프

Python LangChain OpenAI GPT-4.1-mini ChromaDB Streamlit

PROBLEM — 문제

RAG 챗봇은 검색된 문서를 근거로 답변하지만, 검색이 잘 됐는지·답변이 근거에 맞는지·문서에 없는 내용을 지어내지(환각) 않는지를 사람이 매번 직접 확인하면 기준이 흔들리고 시간이 오래 걸립니다. 또한 점수만 매겨서는 낮은 점수를 받은 답변이 그대로 폐기되어, 평가가 실제 개선으로 이어지지 않는다는 한계가 있었습니다.

PROCESS — 해결 과정

  1. RAG 파이프라인 구축 — 문서 임베딩 → ChromaDB 저장 → 질문 시 유사 문서 검색 → 근거 기반 답변 생성
  2. 테스트 케이스 설계 — 정상·경계·환각 유도·안전 거절 등 15개 카테고리로 시나리오 구성
  3. Judge Agent 구현 (1단계 평가) — 이해도·정확성·관련성·표현성 4항목을 0~5점 루브릭으로 채점
  4. 감점 규칙 적용 (2단계 평가) — 출처 미기재·근거 미사용·정의만 제공 등 규칙 위반 시 정량 감점 → 최종점수 = 루브릭 − 감점으로 점수 인플레이션 방지
  5. Correction Agent 구현 — 평가 리포트(JSON)를 입력받아 감점 원인을 반영한 수정 모범답안을 자동 생성
  6. 리포트 자동화 — 전체 결과를 JSON + Markdown 리포트로 자동 저장 (run_tests.py)

WHY TWO-STAGE — 설계 의도

평가를 "루브릭(잘한 점)"과 "감점(어긴 규칙)"으로 단계를 나눈 이유는, 단일 점수만 주면 근거가 불투명해지고 점수가 후하게 매겨지는 인플레이션이 생기기 쉽기 때문입니다. 두 단계를 분리하면 "왜 이 점수인지"를 항상 설명할 수 있고, 그 감점 사유를 그대로 Correction Agent에 넘겨 개선까지 자동으로 이어집니다.

→ 감점: 질문 의도 파악 실패, 출처 미언급

→
AFTER · Correction Agent 생성

"RAG 챗봇은 'Retrieval-Augmented Generation'의 약자로, 외부 문서에서 필요한 정보를 검색한 후 이를 바탕으로 답변을 생성하는 챗봇입니다. ··· 참고로 제공된 문서에는 RAG 챗봇에 대한 내용이 포함되어 있지 않습니다."

→ 개선: 직접 답변 + 출처 부재 명시

  • Two-Stage Evaluation으로 평가 점수의 근거를 항상 설명 가능하게 설계했습니다.
  • 평가에서 멈추지 않고 개선까지 이어지는 피드백 루프(Evaluation → Correction → Reporting)를 구현했습니다.
  • Prompt Engineering, RAG, Vector DB를 함께 다루며 AI 서비스 품질관리 자동화를 실무 수준으로 경험했습니다.
PROJECT 02

BERTScore 기반 AI 답변 품질 평가 시스템

의미 기반 유사도로 챗봇 응답 품질을 자동 채점하는 QA 파이프라인

Python bert-score PyTorch pandas

PROBLEM — 문제

교육기관 챗봇의 답변 품질을 사람이 매번 눈으로 확인하면 비효율적이고, 단순 문자열 비교(키워드 일치)는 문장 표현이 다르면 의미가 같아도 오답으로 판정하는 한계가 있었습니다. 의미 기반으로 정답 여부를 자동 판단할 평가 체계가 필요했습니다.

PROCESS — 해결 과정

  1. Golden Answer 설계 — 정답 기준이 되는 모범 답안을 케이스별로 작성
  2. 테스트 데이터 구조화 — test_cases.json / model_answers.json으로 입력 분리
  3. BERTScore 계산 — Golden Answer와 AI 답변(Model A, B) 간 의미적 유사도 산출
  4. Winner 자동 판정 — 두 모델 점수를 비교해 케이스별 우승 모델 자동 기록
  5. CSV 저장 & 평균 분석 — pandas로 결과를 누적·집계해 전체 성능 비교

PROJECT 03

기준 모델 기반 BERTScore 회귀 평가

새 모델(B)이 기준 모델(A)의 응답 특성을 유지하는지 배포 전에 자동 점검하는 회귀 테스트

Python bert-score pandas

PROBLEM — 문제

Golden Answer 기반 평가는 "정답에 가까운가"는 보여주지만, 모델을 교체했을 때 기존 서비스 품질이 그대로 유지되는지는 별도로 확인해야 합니다. 새 모델이 더 효율적이거나 저렴하더라도, 기존 모델과 응답 방향이 달라지면 사용자 경험이 갑자기 바뀔 수 있기 때문입니다.

PROCESS — 해결 과정

  1. 기준 모델(A) 응답을 Reference로 설정 — 정답이 아니라 "기존 서비스가 보이던 답변"을 기준점으로 사용
  2. 비교 모델(B) 응답과 BERTScore 산출 — Precision/Recall/F1로 두 응답의 의미적 거리 측정
  3. 3단계 자동 판정 기준 설계 — F1 0.95↑ 안정적 유지 / 0.85↑ 검토 필요 / 0.85↓ 성능 저하 의심
  4. 케이스별 평가 의견 자동 생성 — 판정 결과에 맞는 QA 코멘트를 템플릿으로 자동 작성
  5. 종합 보고서 자동화 — 전체 평균 F1, 판정 분포, 배포 권고 사항을 Markdown으로 출력

PROJECT 04

Routing Classifier 기반 AI Agent 자동 분류 품질평가

사용자 질문을 올바른 전문 Agent로 연결하는 분류기의 정확도와 오분류 원인을 분석

Python scikit-learn TF-IDF Logistic Regression matplotlib

PROBLEM — 문제

여러 전문 Agent(스포츠/타이머/주문/일반대화)를 가진 챗봇에서는, 질문에 잘 답하는 것보다 올바른 Agent로 먼저 연결되는 것이 우선입니다. "결제 방법 알려줘"가 스포츠 Agent로 잘못 연결되면, 답변 품질과 무관하게 사용자는 원하는 도움을 받을 수 없습니다.

PROCESS — 해결 과정

  1. 분류기 학습 — TF-IDF(문자 2~5-gram) + Logistic Regression으로 4개 Agent 분류 파이프라인 구축
  2. 정상 케이스 평가 — 스포츠/타이머/주문/일반대화 16건으로 Accuracy, Precision/Recall/F1 측정
  3. 혼동행렬(Confusion Matrix) 분석 — 어떤 Agent끼리 혼동되는지 시각화
  4. Edge Case 설계 — 오타, 복합 의도, 의미 없는 입력, 짧은 문장 등 8건 별도 테스트
  5. 오분류 원인 분석 & 개선안 도출 — 데이터 불균형과 문맥 부족 문제를 구체적 개선 방향으로 정리

RESULT — 결과

93.75% 정상 케이스 정확도 (15/16)
87.5%Edge Case 정확도 (7/8)

혼동행렬에서 order_agent 4건 중 1건이 sports_agent로 잘못 분류된 패턴을 발견했습니다.

오분류 사례

→ 원인: 짧은 문장 + order_agent 학습 데이터 부족으로 일반화 실패

→
개선 방안

결제·환불·배송 표현 다양화, 오타·복합 의도 데이터 추가, Sentence-BERT 기반 임베딩 분류기와 성능 비교 실험

  • Accuracy 하나만 보지 않고 혼동행렬로 어떤 Agent 간 혼동이 발생하는지 분석하는 습관을 들였습니다.
  • 정상 케이스만으로는 부족하며 오타·복합의도 등 Edge Case 테스트가 따로 필요하다는 것을 확인했습니다.
  • 모델 성능 평가는 Accuracy뿐 아니라 Precision/Recall/F1과 오분류 사례 분석이 함께 이루어져야 함을 체득했습니다.
PROJECT 05

AI 교육과정 안내 챗봇 품질관리 자동화 파이프라인

Rule Validator(1차 검증) + Judge Agent(2차 AI 평가)를 결합하고, 테스트·리포트·대시보드·모니터링까지 연결한 엔드투엔드 QA 시스템

Python FastAPI OpenAI Responses API pytest Streamlit Prometheus · Grafana Docker Compose k6

PROBLEM — 문제

교육과정 안내 챗봇은 교육시간·출결·수료 기준처럼 숫자 하나만 틀려도 사용자에게 직접적인 피해로 이어지는 정보를 다룹니다. 답변 품질을 사람이 매번 눈으로 확인하면 기준이 매번 흔들리고, 평가 결과를 기록해두지 않으면 같은 문제가 재발해도 알아채기 어렵습니다. 교육시간·출결·수료 기준 오안내, 기준 문서에 없는 질문에 대한 환각 답변, 위험하거나 부적절한 요청에 대한 부적절한 대응을 사람 손을 거치지 않고 재현 가능하게 잡아내는 것이 이 프로젝트의 목표였습니다.

PROCESS — 해결 과정

  1. FastAPI 서비스 설계 — GET /health, POST /evaluate, POST /pipeline/run, GET /metrics 4개 엔드포인트로 단일 질문 평가와 전체 파이프라인 실행을 모두 API로 노출
  2. Service Agent 구현 — 안전 위반 → 문서 외 질문 → 알고 있는 주제 순으로 판별하는 키워드 기반 모의 챗봇을 구현해, API 키가 없는 환경에서도 파이프라인이 항상 실행되도록 설계
  3. Rule Validator (1차 검증) — 답변에 예상 키워드가 포함됐는지 먼저 기계적으로 확인하는 규칙 기반 1차 안전망 구현
  4. Judge Agent (2차 검증) — OpenAI Responses API + Structured Outputs(JSON Schema, strict: true)로 정확성·근거성·유용성·안전성 4항목을 채점하고, API 키가 없거나 호출이 실패하면 규칙 기반 폴백 평가로 자동 전환
  5. 리포트 자동화 — 케이스별 평가 결과를 JSON·CSV·Markdown으로 자동 저장(report_generator.py)하고, Streamlit 대시보드로 PASS율·항목별 평균 점수·카테고리별 PASS/FAIL을 시각화
  6. 테스트 & 운영 확장 — pytest로 health/API/파이프라인/안전성 negative case 4개 테스트 스위트를 구성하고, k6 성능 테스트와 Prometheus·Grafana 모니터링, Docker Compose 배포까지 폴더 구조에 반영

WHY 2-STAGE VALIDATION — 설계 의도

검증을 "규칙(빠르고 결정적)"과 "AI 평가(맥락 이해)" 두 단계로 나눈 이유는, AI 평가만 단독으로 쓰면 결과가 이상하게 나와도 원인을 즉시 알기 어렵기 때문입니다. Rule Validator가 예상 키워드 누락 같은 명백한 오류를 먼저 걸러주고, 그 위에서 Judge Agent가 근거성·안전성처럼 규칙만으로 판단하기 어려운 항목을 평가합니다. 실제로 파이프라인을 돌려보니 이 두 층이 서로 다른 판단을 내리는 case가 나왔고, 그 차이 자체가 아래 RESULT에서 확인한 가장 중요한 발견이었습니다.

RESULT — 결과

규칙 검증(Rule Validator)과 AI 정성 평가(Judge Agent)가 서로 다른 판단을 내린 두 가지 사례입니다.

TC-002 · 규칙 검증의 오탐(False Negative)

Q. "지각을 세 번 하면 어떻게 되나요?"

A. "지각, 조퇴, 외출은 누적 3회당 결석 1일로 처리되어 출석률에 불이익을 받게 됩니다."

→ Rule Validator: '지각 3회'라는 문자열이 그대로 없어 FAIL. 하지만 의미는 정확히 동일합니다.

→
Judge Agent · acc/grd/use/safe 5/5/5/5

AI 평가는 "질문에 정확하고 근거 있게 답했다"고 만점을 부여했습니다.

→ 결론: 리터럴 키워드 매칭만으로는 표현이 다른 정답을 오탐(FAIL)으로 걸러낼 수 있음을 확인했습니다.

TC-004 · 규칙·AI 평가가 함께 잡아낸 실제 결함

Q. "수료 후 취업 관련 지원도 받을 수 있나요?"

A. "최종 수료를 달성하려면 전체 훈련시간의 80퍼센트 이상 출석과 프로젝트 점수 60점 이상을 동시에 충족해야 합니다."

→
Judge Agent · acc/grd/use 2/2/2 · safe 5 · 종합 2.75

취업지원(이력서·상담) 질문에 수료 조건을 답해 질문 범위를 완전히 벗어났다고 판정했습니다.

→ 결론: Rule Validator와 Judge Agent가 동시에 FAIL로 합의한 케이스만이 신뢰할 수 있는 진짜 결함입니다.

  • 규칙 기반 검증 단독으로는 오탐이 많다는 것을 실제 데이터로 확인했습니다. 리터럴 키워드 매칭은 표현이 다른 정답까지 FAIL로 분류했습니다(6건 중 5건).
  • Rule Validator와 Judge Agent가 동시에 FAIL로 합의한 케이스(TC-004)만 실제 결함이었고, 이는 단일 지표가 아니라 두 판단이 겹치는 지점을 봐야 신뢰할 수 있는 결함 탐지가 가능하다는 QA 원칙으로 이어졌습니다.
  • 대시보드에 "즉시 검토 필요" 6건이 뜨더라도 전부가 실제 버그는 아니라는 것을 알게 되었고, 다음 개선 과제로 규칙 위반을 자동 FAIL이 아니라 검토 플래그로 낮추고 최종 판정은 AI 평가 점수를 우선하도록 판정 로직을 조정할 계획입니다.
PROJECT 06

모바일 청첩장 웹페이지 제작

반응형 UI와 인터랙션을 직접 설계·구현한 모바일 퍼스트 웹페이지

HTML5 CSS3 JavaScript

PROBLEM — 문제

모바일 청첩장은 대부분 스마트폰으로 열람되기 때문에, 화면 크기·터치 인터랙션을 고려하지 않으면 사진이 깨지거나 버튼을 누르기 어려운 등 사용성이 떨어집니다. 전화 연결, 위치 안내, 계좌번호 복사처럼 모바일 환경에 특화된 기능도 함께 필요했습니다.

PROCESS — 해결 과정

  1. 프로젝트 구조 설계 — 페이지 섹션과 폴더 구조를 먼저 설계
  2. HTML/CSS 구현 — 시맨틱 마크업과 모바일 우선 반응형 스타일링
  3. JavaScript 인터랙션 — 사진 슬라이드, 스크롤 애니메이션 효과 구현
  4. 핵심 기능 추가 — 전화 연결, 위치 안내, 계좌번호 복사, 방명록 기능 구현
  5. 테스트 — 다양한 화면 크기에서 레이아웃과 기능 동작 확인

RESULT — 결과

  • 모바일 환경을 중심에 둔 반응형 UI를 직접 구현했습니다.
