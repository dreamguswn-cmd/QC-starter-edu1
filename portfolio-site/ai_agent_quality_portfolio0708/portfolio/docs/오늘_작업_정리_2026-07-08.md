# 오늘 작업 정리 (2026-07-08)

## 1. 작업 목적

오늘 작업의 목표는 기존 "교육과정 안내 챗봇" 프로젝트를 단순 API 구현 수준에서 끝내지 않고, 실제 운영 상황을 가정한 품질관리 포트폴리오로 보강하는 것이었다.

특히 다음 항목을 중심으로 정리했다.

- 장애 상황을 일부러 재현할 수 있는 실습 API 추가
- 장애 API에 대한 자동 테스트 작성
- 입력값 검증과 예외 상황 문서화
- k6 부하 테스트 설정 오류 수정 및 결과 기록
- 초보자도 이해할 수 있는 프로젝트 소개 문서 작성

---

## 2. 장애 실습 API 추가

### 대상 파일

- `app/fault_lab.py`
- `app/main.py`
- `tests/test_fault_lab.py`

### 작업 내용

`/fault-lab` 엔드포인트를 통해 서버 장애 상황을 직접 재현할 수 있도록 구성했다.

지원하는 시나리오는 다음과 같다.

| scenario | 동작 |
|---|---|
| `normal` | 정상 응답 반환 |
| `delay` | 지정한 시간만큼 응답 지연 후 성공 응답 |
| `error500` | 의도적으로 500 서버 오류 발생 |
| `timeout` | 지정한 시간만큼 지연 후 504 타임아웃 오류 발생 |

이 API는 실제 서비스 장애를 흉내 내기 위한 훈련용 기능이다. 포트폴리오 관점에서는 단순히 "서버가 잘 된다"가 아니라, "서버가 실패할 때도 어떻게 관찰하고 기록할 수 있는가"를 보여준다.

### 테스트

`tests/test_fault_lab.py`에 다음 케이스를 추가했다.

- 정상 시나리오가 200 응답을 반환하는지 확인
- 지연 시나리오가 실제로 지정 시간 이상 지연되는지 확인
- 서버 오류 시나리오가 500 응답을 반환하는지 확인
- 타임아웃 시나리오가 504 응답을 반환하는지 확인

## 3. 입력값 검증 보강

### 대상 파일

- `app/schemas.py`

### 작업 내용

`AskRequest` 모델에 Pydantic 검증 조건을 명시했다.

- `question`: 최소 1자, 최대 500자
- `mode`: `rule` 또는 `api`만 허용

이를 통해 빈 질문이나 잘못된 모드 요청은 서버 내부 로직까지 들어가기 전에 422 응답으로 차단된다.

이 작업은 `docs/defect_report.md`의 `DEF-002` 항목과 연결된다.

## 5. k6 부하 테스트 수정 및 결과 기록

### 대상 파일

- `performance/k6_test.js`
- `performance/results/k6_load_test_report.md`
- `performance/results/summary.json`

### 문제

기존 k6 설정에서 `vus` 값이 지나치게 크게 설정되어 로컬 개발 서버가 감당할 수 없는 부하가 발생했다.

그 결과 실제 서버 기능 문제가 아니라 테스트 설정 문제인데도, 서버 장애처럼 보이는 상황이 발생했다.

### 조치

문서 기준에 맞춰 k6 설정을 다음과 같이 정리했다.

```javascript
export const options = {
  vus: 3,
  duration: '10s',
  thresholds: {
    http_req_failed: ['rate<0.05'],
    http_req_duration: ['p(95)<1000'],
  },
};
```

테스트 대상은 `/health` API로 두고, 다음 항목을 확인하도록 구성했다.

- HTTP 200 응답 여부
- JSON 응답의 `status` 값이 `ok`인지 여부
- 응답 시간이 1초 미만인지 여부

### 의미

이 작업을 통해 "부하 테스트 실패"가 항상 "서버 기능 결함"은 아니라는 점을 문서화했다. 테스트 도구의 설정값도 품질관리 대상이라는 것을 보여주는 사례가 되었다.

## 6. 초보자용 프로젝트 소개 문서 작성

### 대상 파일

- `프로젝트_소개.md`

### 작업 내용

프로젝트를 처음 보는 사람이 이해할 수 있도록 다음 내용을 쉬운 말로 정리했다.

- 이 프로젝트가 무엇인지
- 왜 만들었는지
- 전체 구조가 어떻게 되는지
- 각 폴더가 어떤 역할을 하는지
- 챗봇이 답변을 만드는 방식
- 답변 품질을 검사하는 방식
- 실행 방법
- 장애 실습 API의 목적
- 주요 기술 용어 설명

핵심 메시지는 다음과 같다.

> 이 프로젝트는 챗봇 하나를 만드는 것이 아니라, 챗봇의 품질을 개발, 테스트, 리포트, 모니터링, 부하 테스트, 장애 실습까지 관리하는 전체 파이프라인을 보여주는 포트폴리오다.

## 7. 오늘 작업의 포트폴리오 의미

오늘 작업으로 프로젝트의 성격이 더 명확해졌다.

이 프로젝트는 단순한 FastAPI 챗봇 예제가 아니라, 다음 역량을 보여준다.

- AI 답변 품질을 테스트 케이스로 검증하는 능력
- 규칙 기반 검증과 Judge Agent 평가를 함께 사용하는 품질관리 설계
- 장애 상황을 재현하고 테스트하는 운영 관점
- pytest 기반 회귀 테스트 작성
- k6 기반 성능 테스트와 결과 해석
- 결함 보고서를 통해 문제, 원인, 조치, 재발 방지를 기록하는 QA 문서화 능력
- 초보자와 비전공자도 이해할 수 있도록 기술 내용을 설명하는 문서화 능력

## 8. 검증 결과

전체 자동 테스트를 다시 실행해 현재 코드가 정상 동작하는지 확인했다.

```bash
pytest tests/ -v
```

결과는 다음과 같다.

- 총 29개 테스트 통과
- 실패 테스트 없음
- 경고 2건 발생
  - FastAPI TestClient 관련 Starlette deprecation warning
  - `.pytest_cache` 쓰기 권한 관련 PytestCacheWarning

경고는 테스트 실패로 이어지지는 않았으며, 현재 기능 검증 결과는 GREEN 상태다.
