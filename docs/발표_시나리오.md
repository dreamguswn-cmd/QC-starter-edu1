## 슬라이드 2 — 프로젝트 개요 (약 1분)

> 저희가 다룬 대상은 **FastAPI 기반 교육과정 안내 챗봇**입니다.
> 사용자가 교육시간이나 출결 규정을 물어보면 근거 기반으로 답변하고, 범위 밖 질문이나 위험한 질문은 거절하는 서비스입니다.
>
> 품질관리 범위는 네 가지입니다.
> **첫째**, pytest 자동 테스트 25건으로 기능을 검증했고,
> **둘째**, Prometheus와 Grafana로 로그·모니터링 체계를 구축했으며,
> **셋째**, k6로 성능 테스트를 수행했고,
> **넷째**, 발견한 결함 5건을 Jira에 등록해 관리했습니다.
>
> 핵심 메시지를 미리 말씀드리면 — 저희는 **k6 부하테스트 설정 오류로 오류율 100%가 발생하는 장애**를 발견했고, 이를 수정해 **오류율 0%, p95 2.3ms**로 개선했습니다. 이 과정이 오늘 발표의 중심입니다.

**[전환 멘트]** 그럼 테스트를 어떻게 설계했는지 보시겠습니다.

## 슬라이드 3 — 테스트 설계 (약 1분)

> 총 25건의 pytest 자동 테스트를 설계했고, **전체 PASS, Fail 0건**입니다.
> 검증 포인트는 Health Check부터 정상 질문 응답, 빈 질문·형식 오류 같은 Negative 입력, 그리고 **fault-lab이라는 장애 실습 API**까지 포함합니다.
>
> 여기서 강조하고 싶은 것은 장애 시나리오입니다.
> 저희는 500 오류, 응답 지연·타임아웃을 **일부러 재현할 수 있는** fault-lab API를 만들어 테스트했습니다.
> "서버가 잘 된다"가 아니라 **"서버가 실패할 때도 관찰하고 기록할 수 있는가"**를 검증한 것입니다.
>
> 그리고 가장 중요한 장애가 여기서 나옵니다 — **k6의 vus 값이 문서 설계는 3인데 실제 스크립트에는 10,000으로 잘못 설정**되어 있었습니다. 이로 인해 오류율 100%에 좀비 소켓이라는 2차 피해까지 발생했는데, 자세한 내용은 성능테스트 슬라이드에서 말씀드리겠습니다.

**[전환 멘트]** 먼저 모니터링 결과부터 보시겠습니다.

## 슬라이드 4 — 모니터링 결과 (약 1분)

> 화면 오른쪽이 실제 Grafana 대시보드 캡처입니다.
> 요청 수, 성공률, 오류율, 응답시간, p95까지 **네 가지 핵심 지표를 실시간으로 관측**할 수 있게 구성했습니다.
>
> 특히 주목해 주실 부분은 — 관측 기간 중 **품질평가 PASS율이 10%에서 22%까지 상승하는 구간**이 그래프에 그대로 드러난다는 점입니다.
> 또한 fault-lab 장애 시나리오를 실행하면 오류율과 지연 지표가 **동시에 상승하는 것**을 대시보드에서 즉시 확인할 수 있었습니다.
>
> 즉, 문제가 생기면 로그를 뒤지는 것이 아니라 **대시보드에서 먼저 신호를 잡을 수 있는** 구조를 만들었습니다.

**[전환 멘트]** 다음은 오늘 발표의 하이라이트, 성능테스트 결과입니다.

## 슬라이드 5 — 성능테스트 결과 (약 1분 30초) ★ 핵심 슬라이드

> 표를 봐 주십시오. 1차 실행과 2차 실행의 차이가 극명합니다.
>
> **1차 실행**: 총 141,375건 요청에 **오류율 100%** — 응답시간은 측정 자체가 불가능했습니다.
> 처음에는 서버 장애로 보였습니다. 그런데 분석해 보니 **서버 문제가 아니라 테스트 스크립트의 vus 설정이 10,000으로 잘못되어 있던 것**이었습니다. 사실상 로컬 서버에 DoS 공격을 한 셈이었죠.
>
> 그래서 1차는 **"테스트 무효"** 처리하고, 문서 기준값인 vus 3 · 10초로 수정해 재실행했습니다.
> **2차 결과**: 오류율 0.00%, 평균 1.57ms, **p95 2.3ms** — 기준인 1000ms를 크게 상회하며 통과했습니다.
>
> 아래 그래프가 이 극적인 개선을 시각화한 것입니다. 왼쪽 빨간 막대(100% 실패)가 초록 막대(0%)로 바뀐 것이 한눈에 보이실 겁니다.
>
> 여기서 얻은 교훈은 명확합니다 — **"부하테스트 실패"가 항상 "서버 결함"은 아니다. 테스트 도구의 설정값도 품질관리 대상이다.**

**[전환 멘트]** 이 결함을 포함해 저희가 관리한 결함 전체를 보시겠습니다.

## 슬라이드 6 — 결함 및 개선 결과 (약 1분 30초)

> 주요 결함 TOP 3입니다.
>
> **첫째, DEF-004** — 방금 보신 k6 vus 설정 오류입니다. 심각도 High. 수정 후 오류율 0%, p95 4.48ms로 PASS 확인했습니다.
>
> **둘째, DEF-005** — Docker Compose 실행 시 포트 8000 충돌로 App 컨테이너가 생성에 실패한 건입니다. 여기서 중요한 발견이 있었는데, 컨테이너가 "Started" 상태로 떠 있어도 **포트 매핑이 비어 있으면 실제로는 접근 불가능**하다는 점입니다. force-recreate로 완전히 재생성해 복구했습니다.
>
> **셋째, TC-008~010** — 범위 외 질문에서 Rule Validator가 과잉 검증으로 전부 FAIL 처리한 건입니다. 의미상 정확한 거절 답변인데도 키워드가 문구에 없다는 이유로 FAIL이 났습니다. 검증 로직을 개선해 정상 판정되도록 했습니다.
>
> 이 5건의 결함은 모두 **Jira WT4-26~31로 등록해 추적**했고, 현재 전건 Closed 상태입니다.
> 개선 전후를 요약하면 — 오류율 100%에서 0%로, Docker 접근 불가에서 정상 기동으로, 회귀 테스트 전체 PASS까지 확인했습니다.

## 슬라이드 7 — 최종 품질 판정 (약 1분 30초)

> 저희의 최종 판정은 **"조건부 운영 가능"**입니다.
>
> 판정 근거는 네 가지입니다.
> 기능 테스트 25건 전체 PASS, 품질 파이프라인 7/7 PASS.
> 모니터링 — Prometheus 수집과 Grafana 실시간 관측 정상.
> 성능 — k6 오류율 0%, p95 2.3ms로 기준 충족.
> 결함 — 5건 전건 Closed에 회귀 방지 테스트까지 승계했습니다.
>
> 다만 **rule 모드는 기준을 모두 충족**하지만, **api 모드는 외부 LLM 의존**이 있어 지연·비용 정책 확정 후 확대를 권장하기 때문에 "조건부"입니다.
>
> 향후 과제는 세 가지입니다 — RAG 연동으로 커버리지 확대, Golden Answer 이중 검증, 그리고 평가 로그 시계열 적재를 통한 품질 변화 자동 리포트입니다.
>
> 마지막으로 팀 회고를 한 문장으로 요약하면 —
> **"기능 테스트가 모두 PASS여도 서비스 품질이 보장되는 것은 아니다. 성능, 운영 환경, 그리고 테스트 도구 자체까지 검증해야 진짜 운영 가능 여부를 판단할 수 있다"**는 것을 이번 프로젝트로 직접 경험했습니다.
>
> 이상으로 4조 발표를 마치겠습니다. 감사합니다.

## 예상 질의응답 (Q&A 대비)

**Q1. vus 10000 오설정은 왜 발생했나요?**
> 스크립트 파일 내 주석에는 "vus=3, duration=10s"라고 명시되어 있었지만 실제 options 값이 10000으로 달랐습니다. 코드와 문서의 불일치를 리뷰에서 잡지 못한 것이 원인이며, 재발 방지책으로 "실행 결과에 큰 영향을 주는 설정값은 코드 리뷰 시 문서와 대조 확인"을 프로세스화했습니다.

**Q2. 좀비 소켓 2차 피해는 무엇인가요?**
> vus 10000 부하로 uvicorn 프로세스가 강제 종료된 뒤에도, Windows 커널에 8000번 포트 Listen 소켓이 남아 신규 연결을 거부했습니다. 프로세스 목록에는 없는데 소켓 테이블에는 남아 있는 상태였고, taskkill로도 해제되지 않아 8001번 포트로 우회해 재실행했습니다.

**Q3. rule 모드와 api 모드의 차이는?**
> rule 모드는 순수 Python 키워드 매칭이라 ms 단위로 응답하고 외부 의존이 없습니다. api 모드는 외부 LLM을 호출해 답변 품질은 높지만 초 단위 지연과 비용이 발생합니다. 그래서 성능 기준(p95 1000ms)은 rule 모드만 충족 확인된 상태이고, api 모드는 캐싱·하이브리드 라우팅 적용 후 확대를 권장했습니다.

**Q4. TC-008~010 FAIL은 결함인가요, 케이스 설계 문제인가요?**
> 둘 다입니다. 챗봇의 답변 자체는 의미상 정확한 거절이었지만, 검증기가 문구 완전일치를 요구해 FAIL 처리했습니다. 즉 "검증 도구의 결함"이며, 이 경험에서 "검증 도구도 테스트 대상이다"라는 교훈을 얻었습니다.

**Q5. 모니터링 지표 중 PASS율이 10→22%로 오른 이유는?**
> 관측 기간 중 품질 파이프라인 개선(Rule Validator 로직 수정)이 반영되면서 평가 통과율이 상승한 구간입니다. 개선 활동이 지표에 실시간으로 반영되는 것을 확인한 사례입니다.
